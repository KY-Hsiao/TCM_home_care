# 專案 AGENTS.md

## General working style

- 回覆與文件內容預設使用繁體中文，除非專案內另有明確要求。
- 先閱讀現有規格、註解與檔案結構，再提出修改。
- 優先做小範圍、可驗證、可回退的修改。
- 若規格與程式碼不一致，先指出不一致處，再提出修正方案。
- 不要任意更改核心名詞、按鈕名稱、欄位名稱或輸出檔名格式，除非規格明確要求。
- 新增功能時，優先維持可讀性、可維護性與穩定性。

## Coding rules

- 避免一次大幅重寫整個檔案，除非使用者明確要求。
- 變更前先確認會受影響的函式、資料流與輸出格式。
- 任何涉及資料切分、快取、檔案輸出、模型結果的修改，都要檢查是否會造成前後不一致。
- 若有邊界條件、失敗條件或 fallback 規則，需明確寫入程式與註解。

## Documentation rules

- 規格文件更新時，優先保持步驟編號、名詞與欄位名稱一致。
- 若新增流程，應同步補上錯誤處理、輸出內容與檔名規則。
- 若方法為近似作法，應明確標示限制，不要假裝成完全理論解。

## Collaboration rules

- 以單一 thread 連續完成同一組需求，避免在多個 thread 間切換造成重複確認。
- 編輯前先整體收斂修改範圍，再集中套用，避免連續多次小 patch。
- 同一批需求若涉及多個檔案，優先合併成一次較完整的修改提交，而不是逐檔零碎套用。
- 若只是清理文案、命名一致性或測試同步，優先合併成單一批次完成。
- 除非遇到高風險衝突或需求尚未明確，否則不要為中間小幅變更反覆停下來等待確認。
